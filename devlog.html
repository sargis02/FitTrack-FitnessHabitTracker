Tracker Planning Dev Log

Development Log

Phase 1: Activity Form + Display

Day 1

Built: The main HTML structure including header, navigation bar, and card-based layout.

Reason: I wanted to first create a clear structure before adding any functionality so that everything would be organised from the start.

Reflection: This helped me understand how to divide a web app into sections (form, stats, list, etc.). I also decided early to use a card layout because it makes the UI easier to read and expand later.

Phase 2: Delete Functionality

Day 2

Built: A form to add activities and a function to display them dynamically on the page.

Challenge: I realised that user input could break the layout if it included special characters.

Solution: I created an escapeHtml() function to safely display user input.

Reflection: This was my first time thinking about security and not just functionality. It made me realise that even simple apps need to handle user input carefully.

Phase 3: localStorage Save/Load

Day 2

Built: Switched from in-memory activities to localStorage. Now activities stay even after refreshing the page.

Challenge: localStorage only saves strings and Data could break if JSON parsing fails or storage is full.

Solution: I had to use JSON.stringify() and JSON.parse(). I also added try/catch so the app won’t crash if saved data is broken.

AI Use: I used AI assistance to better understand how to safely handle errors with localStorage.

Reflection: This made the app feel more like a real application instead of something that resets every refresh.

Phase 4: CRUD Operations (Edit/Delete)

Day 3

Built: Edit functionality (prefilling form and updating data) and delete buttons for activities.

Challenge: Activities are created dynamically, so adding event listeners to each one individually became messy.

Solution: I used event delegation by adding a single listener to the parent container.

AI Use: I researched best practices for handling events on dynamic elements and found that event delegation is the most efficient approach.

Reflection: This was an important learning moment because I understood how to handle dynamic elements efficiently.

Phase 5: Statistics & Date Calculations

Day 4

Built: Statistics such as total activities, weekly count, monthly count, and average duration.

Challenge: JavaScript considers Sunday as the start of the week, but I wanted Monday.

Solution: I wrote a custom getStartOfWeek() function with a special case for Sunday.

AI Use: I used AI and online references to confirm the logic and edge cases.

Reflection: This helped me understand how tricky date handling can be and how important edge cases are.

Phase 6: Filtering & Search

Day 5

Built: Filtering by type and time period, search by activity name.

Design Decision: I combined all filtering logic into one function so the app always renders from a single source.

Challenge: I had to ensure that all filters work together (e.g. search + type filter at the same time).

Solution: I implemented a comprehensive filtering function that applies all selected filters sequentially.

AI Use: I used AI to help me structure the filtering logic in a way that is efficient and easy to maintain.

Reflection: This phase improved my understanding of how data and UI should stay in sync.

Phase 7: Streaks, Goals, Chart & UI Enhancements

Day 6

Built: Streak counter, goal tracking system, progress bars, and a weekly activity chart.

Design Decision: I matched goals with activities using name comparison to keep the system simple and flexible.

Challenge: The layout became crowded on smaller screens as more features were added. The streak counter was not updating correctly if the user logged yesterday but not today.

Solution: I used responsive CSS (auto-fit and flexible grids) to improve layout. I adjusted the logic to check both today and yesterday before continuing the streak calculation.

AI Use: I used AI to help me in designing the logic of Streak Counter and also to find the best way to implement a simple progress bar using CSS.

Reflection: I also chose to build the chart using pure CSS and JavaScript instead of libraries to keep the project simple and lightweight.

Phase 8: Weekly Bar chart & Statistics Dashboard

Day 6

Built: A weekly bar chart and a statistics dashboard showing total activities, weekly total, monthly total, average duration, and counts of workouts vs habits.

Design Decision: I decided to build a simple bar chart and statistics dashboard to provide users with a quick overview of their activity patterns and progress towards their goals. I chose to display both total counts and averages to give a more comprehensive view of the user's habits.

Challenge: To make the weekly bar chart without any libraries and to ensure that the statistics dashboard updates correctly with all the new metrics.

Solution: I used CSS grid to create a simple bar chart where each bar's height is determined by the number of activities on that day. For the statistics dashboard, I updated the existing statistics function to calculate and return the new metrics (workouts and habits) and then updated the rendering logic to display them.

Reflection: I found that building the chart and dashboard from scratch was a great learning experience as it forced me to think about how to structure data and use CSS creatively. It also made me appreciate the power of libraries, but I’m glad I built it myself for this project to keep it simple and lightweight.

Phase 9: Export JSON/CSV & Final Touches

Day 7

Built: Export JSON and Export CSV features to allow users to download their activity data in a structured format.

Design Decision: I decided to implement both JSON and CSV export options to cater to different user preferences and use cases. JSON is great for users who want to keep a structured backup or use the data in other applications, while CSV is more accessible for users who want to view their data in spreadsheet software like Excel or Google Sheets.

Challenge: To create a clean and user-friendly way to trigger the export functions and ensure that the exported files are properly formatted.

Solution: I added two buttons for exporting data, one for JSON and one for CSV. For the JSON export, I used `JSON.stringify()` to convert the activities array into a JSON string and created a downloadable file using a Blob. For the CSV export, I constructed a CSV string by iterating over the activities and formatting each activity's properties as comma-separated values, then created a downloadable file in the same way as the JSON export.

AI Use: I used AI assistance and online resources to implement a simple CSV escaping logic that wraps values in quotes and doubles internal quotes. This way the export works correctly even with complex text inputs.

Reflection: Adding export functionality was a great way to round out the project and provide users with more control over their data. It also gave me experience with working with files in JavaScript, which is a useful skill for many applications.
